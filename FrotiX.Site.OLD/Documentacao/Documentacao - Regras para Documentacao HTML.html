Documentacao - Regras para Documentacao HTML

Guia consolidado a partir de Documentacao em Html.md, Indice Geral e paginas A4 existentes.

Objetivo Geral
  • Converter todos os arquivos *.md de Documentacao/ e subpastas em versoes *.html A4, preservando os originais.
  • Produzir um Manual Tecnico completo e visualmente consistente do FrotiX, com linguagem acessivel e prosa leve.
  • Garantir que cada pagina HTML possa ser impressa ou convertida para PDF sem quebra de layout.
Qualidade > velocidade
Nunca apagar .md
Atualizar documentacao periodicamente
Local e nomenclatura do HTML
  • Gerar cada HTML na mesma pasta do .md origem, sem mover para outras pastas.
  • Nome base = nome real do .md; aplicar sufixo A401, A402... apenas quando o mesmo arquivo precisar de multiplas folhas A4. Nao inventar titulos ou pastas (ex.: nao criar "GESTAO" se nao existir).
  • Converter em ordem alfabetica por diretorio e arquivo, preservando todos os .md.
  • Se o .md original estiver fora do padrao, alinhar o HTML ao nome correto do item documentado.
  • Listar no console o nome e o caminho de cada HTML concluido para rastreabilidade.
Fase 1 - Pesquisa e Aprendizado
  • Ler todos os .md em Documentacao/ e subpastas.
  • Mapear: estrutura de banco (tabelas, views, triggers, stored procedures), padroes de nomenclatura (<Tabela>Id), controllers, Razor Pages, arquivos JS e interacao AJAX.
  • Identificar uso de controles (Syncfusion, Telerik), padroes de inicializacao e comunicacao JS-Controllers.
  • Rever sistemas globais: alertas (alerta.js, sweetalert_interop.js), tooltips/toasts (global-toast.js, syncfusion_tooltips.js), agenda (FullCalendar, recorrencias, viagens), alertas com sino (SignalR).
  • Extrair identidade visual de wwwroot/css/frotix.css (cores, botoes, modais, tela de espera) e exemplos de EndPoints/*.html.
  • Registrar insights em Manual/manual-mockup.md antes de iniciar conversoes.
Fase 2 - Conversao MD -> HTML (A4)
  • Ordem alfabetica por diretorio; criar um .html para cada .md, sem remover os originais.
  • Nomenclatura: (Diretorio) NomeA4XX.html (ex.: (Controllers) HomeControllerA401.html). Se exceder 1 pagina A4, use A4.01, A4.02 etc. para o mesmo arquivo.
  • Layout A4: header laranja #b66a3d com borda dupla (preto + branco), cards brancos, snippets em #33465c.
  • Cores padrao: vinho #722F37, azul #325d88, terracota #A97B6E, verde #557570; variantes "light" em uso complementar.
  • Icones: FontAwesome Duotone; sempre inline (SVG) para exportar PDF; uso intensivo e intencional em headers, cards, listas e notas.
  • Conteudo: prosa leve, mas tecnica; explicar interdependencias (Models -> Controllers -> Pages -> JS -> BD).
Fase 3 - Manual Tecnico
  • Organizar por temas (BD, Razor Pages, Controles, JS, Controllers, Padroes de Codigo, Sistemas Globais, Design System).
  • Incluir termos tecnicos, jargoes e snippets de forma natural; referenciar os HTMLs gerados.
  • Formato: HTML em Manual/ com indice navegavel; manter o mesmo padrao visual A4.
Requisitos Visuais e Estruturais
  • Header: laranja #b66a3d, fonte branca pesada, borda preta fina + borda branca larga.
  • Cards: sombra leve (var(--shadow)), cantos 14px; fundo cinza claro geral (var(--cinza)).
  • Snippets: fundo #33465c; fonte clara; bloco com bordas suaves.
  • Botoes e modais: seguir frotix.css (btn-azul, btn-vinho, header laranja, modal padrao FrotiX).
  • Tela de espera: overlay escuro com logo pulsante (padrao FrotiX Spin Overlay).
  • Tipografia: usar familia padrao do site (Segoe UI/Inter/system). Evitar fontes externas.
Conteudo Minimo por Pagina
  • Contexto e proposito do arquivo convertido.
  • Interdependencias: quem usa, onde, como se conecta (Controllers, Pages, JS, BD).
  • Principais endpoints/metodos ou entidades (quando aplicavel) e fluxo resumido.
  • Padroes de inicializacao (controles Syncfusion/Telerik) e chamadas AJAX.
  • Erros e cuidados: tratamento com Alerta.TratamentoErroComLinha, validacoes e bordas de caso.
Tom narrativo + precisao tecnica
Sem copiar/colar cru de .md: sempre enriquecer
Procedimento Operacional
  • Antes de converter: concluir leitura (Fase 1) e registrar insights no manual-mockup.md.
  • Converter em ordem alfabetica; se faltar espaco em uma A4, criar A4.01, A4.02 para o mesmo item.
  • Validar impressao/PDF: checar se cabe em A4 sem cortes e se icones inline aparecem.
  • Reportar progresso a cada 10-15 arquivos; manter consistencia visual.
  • Nao excluir nem sobrescrever .md; apenas adicionar .html pares.
Pontos Criticos (Regra Critica)
  • Pesquisa completa antes da primeira conversao; sem pular a fase de aprendizado.
  • Visual uniforme: header laranja FrotiX, cinza claro de fundo, cards com icones Duotone, snippets em azul petroleo.
  • Interdependencia explicita: sempre descrever quem consome e quem prove dados.
  • Sem atalhos de velocidade: priorizar texto fluido e explicativo, sem perda de detalhes tecnicos.
  • Conferir nomes: nao gerar HTML para arquivos inexistentes; manter alinhamento ao nome do .md.
Paleta e componentes FrotiX
Consulte sempre GEMINI.md + Claude.md + frotix.css + frotix.js
  • Vinho #722F37 (cancelar/fechar, btn-ftx-fechar)
  • Azul #325D88 (btn-azul, botao primario, tooltips)
  • Terracota #A97B6E (headers suaves, cartoes)
  • Verde #557570 (estados ok suaves)
  • Laranja header #A0522D (btn-header-orange)
  • Botoes padrao: btn-header-orange (novo), btn-azul btn-submit-spin (salvar), btn-ftx-fechar (cancelar), btn-voltar (voltar). Colunas de Acao com icones duotone no tamanho var(--ftx-action-icon).
  • Modal padrao FrotiX: cabecalho com icone duotone, borda dupla branca/preta, fundo claro; usar classes do frotix.css.
  • Overlay/Tela de espera: usar ftx-spin-overlay e logo pulsante (padrao frotix.js); obrigatorio em operacoes assincronas.
  • Badges de status: ATIVO / PENDENTE / ERRO; usar variaveis globais do CSS.
  • Icones duotone em SVG inline: uso intensivo e decorativo/explicativo em titulos, cards e listas; primaria #ff6b35, secundaria #6c757d (ajustar conforme contexto).
